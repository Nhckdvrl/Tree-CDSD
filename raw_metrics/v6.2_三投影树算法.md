# LDT-v6.2：干净三投影树算法说明

## 这版结果对应的模型

当前核心 full500 结果来自 **Qwen3.5-9B**，使用 Qwen2.5-32B 作为 LLM-judge。

也就是说，`analysis/ldt/v5/v62_q35_full500_summary.csv` 里的主表不是多模型平均，而是：

- 推理模型：`qwen3.5-9b`
- judge 模型：`qwen2.5-32b-judge`
- 数据规模：四个多跳数据集各 500 条
- 方法：`ldtv5`，tag 为 `v62_triproj_terminalfix`

## 设计目标

这一版的目标不是继续堆 benchmark-specific 规则，而是在保留“树推理”核心思想的前提下，形成一个干净、可解释、可消融的算法结构。

核心原则：

1. 计划树只负责提出证据路径和约束，不直接提交最终答案。
2. 最终答案只能来自真实跑过的推理树终端叶节点。
3. 不依赖某个数据集的特定模板修补，而是用通用的树投影、终端选择和一致性审计来纠偏。
4. 每个模块都有明确作用，后续可以单独消融。

## 算法主线

对每个问题，LDT-v6.2 会维护并使用四类树结构信息。

第一，证据计划树。

每个 agent 先生成一个 evidence plan，包括：

- 问题要求的答案槽位；
- 需要验证的 bridge chain；
- 可能相关的 passage title；
- 候选答案和置信度。

这些计划被合并为一个 resolved evidence plan，并写入树中。但注意，计划树里的候选答案不会直接进入最终答案候选池。

第二，计划验证。

resolved plan 会被当成一个“待验证假设”，由模型检查其 bridge chain 是否真的被选中 passages 支持。验证结果只作为后续选择依据，不再像早期版本那样硬性关掉 focused tree。

第三，三棵推理投影树。

同一个问题会跑三种真实推理树：

1. `full_tree`：完整上下文推理树。
2. `focused_evidence_tree`：只保留计划选中的证据 passages，但不注入 bridge-chain 约束。
3. `focused_contract_tree`：在 focused evidence 基础上额外注入 answer slot 和 bridge chain 约束。

这样做的目的，是把三种错误模式分开：

- full context 容易被 distractor 干扰；
- evidence-only focused tree 能减少干扰，但可能缺少关系约束；
- contract focused tree 能强化关系链，但如果 plan 错了，也可能被带偏。

第四，终端叶选择。

最终只在三棵真实推理树的 terminal leaves 中选择答案。选择器会检查：

- 是否回答了问题；
- answer type 是否匹配；
- passage support 是否充分；
- 是否是最小答案 span；
- full/focused/contract 三种投影之间是否存在一致性或冲突。

## 关键模块

### 1. 计划不直接提交答案

早期 v5.0 允许 plan answer 作为终端答案，结果计划幻觉会直接覆盖真实推理树。v6.2 去掉了这一点。

现在 plan 只能影响 focused context 和 contract tree，不能绕过推理树直接赢。

### 2. 三投影树

v6.2 的核心是三投影：

```text
原问题
 ├── full-context reasoning tree
 ├── focused evidence-only reasoning tree
 └── focused plan-constrained reasoning tree
```

三棵树都是真实推理树，不是 prompt 后处理。它们各自生成 terminal answer，最后再统一选择。

### 3. 投影一致性审计

如果两个投影给出同一个答案，另一个投影给出不同答案，会额外触发 projection-agreement audit。

这个模块主要修两类问题：

- hop depth 错误，例如只答到父节点，没有答到父亲的父亲；
- full tree 被 distractor 带偏，而两个 focused 投影一致给出更合理答案。

后续 v6.2 加了保护，避免该审计把一个已经正确的 plan-constrained terminal 拉回复合答案或 non-answer。

### 4. 终端 span 清理

只在终端阶段做两类很小的 span 操作：

- 如果当前答案是 `A and B`，而 focused leaf 唯一给出严格子答案 `A`，则允许选择子答案；
- 如果当前答案是单 token，如 `Miquette`，而同一棵树中已经有唯一更完整候选 `Miquette Giraudy`，则扩展为完整实体。

这不是数据集规则，而是通用的终端 span 投影。

## 剩余问题

目前剩余失败主要集中在：

1. 多个 award / attribute 同时出现时，如何选择 benchmark 期望的那个属性；
2. birthplace/location granularity，例如 landmark、municipality、district 的边界；
3. passage 中有隐式支持，但 validator 过于严格；
4. alias、日期、行政区划名称的 judge 敏感问题。

下一步更像算法方向的改进是 terminal candidate lattice：

把每个树终端答案拆成若干 typed candidate span，再让 selector 在 span lattice 上选择，而不是只在整句 terminal answer 之间选择。
