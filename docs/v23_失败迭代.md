# v23 自适应角色边辩论树设计：屎

## 1. 当前定位

v23 不是把 LDT 做成普通 Tree-of-Thought 搜索，而是继续服务“多 agent 辩论”的主线：树是辩论状态的载体，节点是 agent 对局部推理边的立场，边是可被质询、反驳、重开和保留的推理关系。

当前主版名称建议：

```text
Adaptive Role-Edge Debate Tree, ARDT / LDTD-v23
```

它相对 v8 的核心变化不是增加工程兜底，而是把真实轨迹里反复出现的三类失败写成算法组件：

1. 强关系方向题需要证明 role edge，不能只复述目标关系。
2. 被反方打开的高分分支，不能被早停直接压掉。
3. 终局同分时，完成更多桥接边的 terminal path 优先展示给 final debate。

## 2. 算法脉络

整体流程：

```text
多 agent 分层提出 atomic next-hop
→ 合并等价候选
→ 判断 compatible / uncertain / conflict
→ 对冲突或不确定候选做局部 debate
→ 对 final 候选做 terminality debate
→ 保留 contested terminal，同时把被质疑节点重开
→ 对强关系方向题启用 role-edge 审计和 open-frontier patience
→ 最终对 terminal paths 做 path-level debate
→ yes/no 普通冲突用树上 terminal stance 做加权投票
```

v23 的关键点是“自适应”：不是所有题都上最严格的审计。只有 clean question 中出现家谱/姻亲/父系母系/配偶/比较二选一等强关系方向信号时，才启用 strict role-edge 和 open-frontier patience。普通实体桥接题保留 v8/v9 更稳的宽松终止性。

## 3. 组件说明

### 3.1 Role-Edge Terminality

强关系方向题中，terminality debate 不只判断答案类型，还要检查：

- `required_edges`：问题要求的最小关系边；
- `observed_edges`：当前路径真实证明的关系边；
- `role_direction_ok`：方向是否正确；
- `unsupported_assertion`：是否只是复述目标关系。

例如 father-in-law 题中，`spouse` 只是桥实体，答案必须是 `spouse's father`。

### 3.2 Open-Frontier Patience

真实轨迹中出现过：正确反方分支已经在树上出现，但错误 final 分支分数高，导致第 2 层提前停止。v23 只在 strict role-edge 题上启用 open-frontier patience：

```text
若高分 final 存在，但同分/近分 open rival 仍未展开，则暂缓 early stop。
```

这让反方有机会把 “Rupert III” 继续展开成 “Rupert III was the father of John”。

### 3.3 Bare-Entity Guard

strict role-edge 题里，只有一个裸实体名不能关闭路径，除非前缀已经证明它占据目标角色。这个约束避免 “Rupert III” 在还没证明父亲边时被过早当作 final。

### 3.4 Granularity Guard

当问题直接问 “which borough/county/province/state/country...” 时，候选节点必须证明该行政层级，不能把出生地 town 直接当成 borough。这个组件修复了 MuSiQue 中 Whiston/Knowsley 的回归。

### 3.5 Binary Stance Vote

普通 yes/no 题中，如果 final debate 选择了单一路径，但树上保留了多条相反 terminal stance，v23 用 terminal path score 做加权投票。比较型 yes/no（如 more/less than）不启用该投票，避免覆盖路径级比较裁决。

## 4. full50 阶段结果

### 4.1 qwen3-8B

| 方法 | HotpotQA | 2Wiki | MuSiQue | StrategyQA | Macro |
| --- | ---: | ---: | ---: | ---: | ---: |
| v8 contested terminal local | 78 | 78 | 64 | 90 | 77.5 |
| v9 role-contract local | 76 | 80 | 64 | 90 | 77.5 |
| v23 local | 76 | 82 | 64 | 90 | 78.0 |
| v8 contested terminal 32B judge | 82 | 80 | 62 | 90 | 78.5 |

结论：v23 在 qwen3-8B 上主要提升 2Wiki，MuSiQue 持平，Hotpot local 低于 v8 但 judge 仍为 80。宏平均 judge 达到 79.0，和 clean-project 中 qwen3-8B 最强 CDSD 宏平均约 79.0 持平；树方法在 2Wiki judge 上更强。

| 方法 | HotpotQA | 2Wiki | MuSiQue | StrategyQA | Macro |
| --- | ---: | ---: | ---: | ---: | ---: |

结论：qwen2.5-14B 是当前 v23 最有说服力的模型：local 和 judge 都明显优于 v8，也超过 clean-project 中 qwen2.5-14B 的 CDSD 宏平均 78.5。

### 4.3 qwen3-4B 边界

| 模型 | 方法 | HotpotQA | 2Wiki | 说明 |
| --- | --- | ---: | ---: | --- |
| qwen3-4B | v8 local | 72 | 70 | 已有下限观察 |
| qwen3-4B | v23 local | 70 | 68 | 暂不适合作为 4B 默认 |

结论：v23 的 role-edge/terminality 负担对 4B 偏重，小模型更容易在终点审计和最终裁决里被长 prompt 干扰。论文里应把 4B 写作边界，而不是强行宣称全尺寸通用最优。

## 5. 与 CDSD 的关系

当前更合理的论文叙事不是“树方法全面替代 CDSD”，而是：

```text
CDSD 擅长链式中间冲突定位；
ARDT/v23 擅长保留树上反方分支，并在角色方向或终点粒度错误时继续展开。
```

已有 paired flips 显示：

- 树赢 CDSD：CDSD 保守输出 “not provided” 时，树通过 contested terminal/open branch 继续补桥，能得到 Rabbit Hole producer spouse、Pine Springs county capital 等具体答案。
- CDSD 赢树：链式角色方向特别长时，CDSD 的结构化 chain 更容易守住 father/mother/grandparent、record label、league version 等精确边。
- v23 缩小了其中一类差距：father-in-law 错例中，树现在能保留 Rupert 分支并最终选中 spouse's father。

因此后续 case analysis 可以把两者写成互补：

| 情形 | 更适合 CDSD | 更适合 ARDT/v23 |
| --- | --- | --- |
| 线性多跳链、每一步角色明确 | 是 | 可以，但不一定最优 |
| agent 中间事实冲突集中在某一段 | 是 | 局部树辩论也能处理，但成本更高 |
| 早停/终点粒度错误 | 一般 | 是 |
| “信息未给出”但树上仍有可展开分支 | 一般 | 是 |
| 4B 小模型 | 较稳 | 当前偏重 |

## 6. 关键成功轨迹

2Wiki father-in-law：

```text
问题：Who is Catherine ...'s father-in-law?
v9/v20 错误：John, Count Palatine of Neumarkt
v23 正确：Rupert III of the Palatinate
```

树上过程：

1. 第一条路径提出 John 是 Catherine 的 husband。
2. 错误 terminal 把 husband 当 father-in-law。
3. open-frontier patience 保留并展开 Rupert 分支。
4. bare-entity guard 阻止 “Rupert III” 作为裸实体过早关闭。
5. 后续边证明 Rupert III was the father of John。
6. final role-edge audit 选择 spouse's father，而不是 spouse。

这个 case 很适合作为论文主图：它不是多 thought 搜索，而是反方分支在树上获得继续辩论和展开的权利。

## 7. 仍然失败的模式

1. MuSiQue 的版本/时间事实仍不稳，例如 league、record label 等需要更强证据版本选择。
2. qwen3-4B 上 prompt 和终点审计偏重，收益不明显。
3. qwen3-8B 上宏平均只与最强 CDSD 打平，主要优势集中在 2Wiki；若写主表，应强调互补性或 qwen2.5-14B 的显著提升。

## 8. 下一步

1. 跑 v23 full500：优先 qwen2.5-14B 和 qwen3-8B。
2. full500 后做 tree vs CDSD paired case：
   - 树赢：终点粒度、反方展开、non-answer recovery；
   - CDSD 赢：线性角色链、证据版本选择。
3. 做消融：
   - 关闭 role-edge；
   - 关闭 open-frontier patience；
   - 关闭 bare-entity guard；
   - 关闭 granularity guard；
   - 关闭 binary stance vote。
4. 若 full500 中 MuSiQue 仍弱，下一版不要加工程兜底，而应研究“evidence-version edge debate”：让树边记录时间/版本/所属范围。
