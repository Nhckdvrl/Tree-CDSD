# 当前版本问题审计

## 0. 当前 agent 设定

当前不是多模型 agent。实验中通常是同一个基础模型，配 3 个不同 persona 和不同温度：

- careful solver
- verifier / skeptical analyst
- alternative-path solver

Qwen3 系列还加了 `/no_think`，避免 thinking mode 带来不一致。soft personas 常用温度是 `[0.6, 0.6, 0.8]`，普通 personas 常用 `[0.7, 0.7, 0.9]`。

merger、relation judge、terminality referee、final path referee 通常也是同一个 client 的低温调用。它们已经有 moderator 的雏形，但目前没有明确抽象成独立的控制智能体。

## 1. Atomic next-hop 划分质量

质量比较混合。

好的一面：

- v8/v23 能在 Caroline LeRoy spouse -> child、Whiston -> borough、Rupert father-in-law 这类题上形成可读的桥接路径。
- 树节点确实记录了每层 agent proposal、候选合并、terminality debate 和 final path debate。

主要问题：

- 有些 proposal 不是 atomic next-hop，而是把两跳压进一个节点。
- Fletcher Webster 轨迹中，两个 agent 提出“Caroline LeRoy's spouse is Daniel Webster”，另一个 agent 提出“spouse is Daniel Webster and child is Fletcher Webster”。LLM merge 把它们合成了同一个更长节点。这次结果正确，但结构上破坏了 atomic next-hop。
- 这说明下一版不该只靠普通 merge，应先做 **atomicity repair**：如果候选包含多条边，moderator 必须拆成多个 typed edges，再进入合并。

## 2. 合并等价候选质量

当前合并不非常可靠。

已经存在的优点：

- deterministic normalize 可以合并编号、大小写、轻微格式差异。
- LLM merge 能合并大量同义候选。

风险：

- over-merge：把“中间桥接实体”和“桥接实体加最终答案”合并，导致节点粒度变粗。
- under-merge：同一实体的别名、全称/简称、行政区划正式名和短名可能没有合并。
- relation-bearing claims 很难只靠文本相似判断，必须看它们是否表达同一条 typed edge。

建议下一版引入显式 **Tree Moderator / Graph Controller**：

1. 先把每个 proposal 解析成 typed edge。
2. 再按 `(subject, relation, object, answer-slot)` 合并。
3. 对不确定合并交给 LLM merger，但 merger 必须输出 edge-level rationale。

## 3. compatible / uncertain / conflict 判断质量

当前判断也是局部可用但不稳定。

`compatible` 最容易出问题，因为“两个事实都真”不等于“它们属于同一条推理链”。这和 RAG 里的语义相关性问题类似：文本相关不代表支持同一个 answer slot。

建议把 merge、relation、terminality 放到同一个 Tree Moderator 下，形成统一的状态管理：

```text
proposal → atomic edge parsing → edge merge → relation classification → conflict queue
```

这样 conflict 不再只是字符串级竞争，而是“同一 answer slot / relation slot 下的互斥边”。

## 4. 局部 debate 是否真的有效

有效，但触发不够稳定。

有效样例：

- v23 father-in-law 题能把 John 和 Rupert 之间的角色方向问题暴露出来，最终选 Rupert。
- v8 terminality debate 能纠正 spouse/child、town/borough 这类过早终止。

不足：

- 如果 merge 已经把多个 hop 合成一个节点，local debate 后面很难再恢复原始粒度。
- 如果所有 agent 假共识，adversarial challenger 有时找不到有效反分支。

所以 debate 的质量上限被节点划分和合并质量限制。

## 5. terminality debate 是什么意思

terminality debate 是“这个节点是否真的可以作为 final answer”的辩论。

它检查：

- 问题要求的 target slot 是什么；
- 候选答案实际满足的 candidate slot 是什么；
- 当前路径是否只到达中间实体；
- 还缺不缺下一跳。

例如问题问 “child of Caroline LeRoy's spouse”，候选如果只到 “Daniel Webster”，terminality debate 应判定还缺 child hop。

## 6. contested terminal 和重开是什么意思

如果某个 final 被 terminality debate 质疑，v8/v23 不直接删除它，而是：

1. 把它作为 contested terminal branch 保留，降低权重；
2. 同时把原分支视为未完成，继续展开；
3. 最终 path-level debate 可以复审 contested branch。

这比硬删除更符合 debate 思想：败方立场没有被擦除，而是带着质疑记录进入终局裁决。

## 7. role-edge、open-frontier、binary vote 是否太像兜底


当前 v23 的这些组件来自真实失败 case，但目前更像 v23 guarded variant，而不适合放进主算法流程：

- role-edge audit 针对强关系方向题有效，但会引入题型触发条件；
- open-frontier patience 能救一些早停错例，但也会增加成本和噪声；
- granularity guard 对 borough/county 等有用，但容易被看作 benchmark-specific；
- yes/no terminal stance vote 是答案类型聚合，不是树辩论的核心。

还是得从 v8 的这个比较干净的继续去推。

## 8. path-level debate 是什么意思

path-level debate 是最终阶段让 agents 比较完整 terminal paths，而不是只看最后一个答案字符串。

它应该回答：

- 哪条路径完整覆盖了问题关系；
- 哪条路径停在中间实体；
- 哪条路径虽然答案看起来合理，但证据链不完整；
- contested terminal 是否应该被恢复。

它相当于终局 adjudication。

## 9. v62 怎么回归多 agent debate 主线

v62 可以命名为 **Tri-Projection Evidence Debate Tree**。

它不是普通三棵树投票，而是：

1. 多个 agent 生成 evidence plans，提出不同证据路线；
2. resolved evidence plan 是共享议程；
3. plan validation 是证据交叉质询；
4. full/evidence/contract 三投影分别代表开放上下文、证据聚焦、关系约束三种论证视角；
5. terminal selector 是 adjudicator；
6. projection agreement audit 是带 evidence guard 的一致性检查；
7. terminal span cleaning 是 verdict normalization。

这样 v62 就能服务论文题目，而且它的 full500 表现目前最好。
