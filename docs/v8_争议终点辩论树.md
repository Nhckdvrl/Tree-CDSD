# v8 争议终点辩论树设计

## 1. 研究主线

本版重新把算法主线收回到“多 agent 辩论”，树只是辩论发生和被记录的结构载体。

核心过程不是 Tree-of-Thoughts 式的多 thought 搜索/打分，而是：

```text
多 agent 提出局部 next-hop
→ 合并等价候选并识别 conflict / uncertain / compatible
→ 对局部冲突做多 agent debate
→ 对声称 final 的节点做 terminality debate
→ 若 final 有争议，不删除该立场，而是保留 contested terminal branch
→ 原节点作为中间节点继续扩展
→ 最终对 terminal paths 做 path-level debate
```

关键区别：ToT 的核心是“生成候选思路并搜索”；v8 的核心是“agent 对树节点立场进行质询、裁决、降权、保留证据，并把裁决反写到树结构中”。

## 2. 当前主算法

方法名：`ldtd`

### 2.1 Layerwise Proposal

每一层中，每个 agent 只提出一个 atomic next-hop。节点可以是桥接事实，也可以声称 final answer，但 final 不是直接接受，而要进入后续辩论。

### 2.2 Local Relation Debate

同一父节点下的候选会先被合并和关系判断：

- `conflict`：互斥或竞争分支；
- `uncertain`：关系不清或证据不足；
- `compatible`：可共存但不一定应合成；
- `equivalent`：同一局部 hop。

`conflict` 和 `uncertain` group 会触发局部多 agent debate，裁决写回候选节点历史。

### 2.3 Terminality Debate

这是 v8 的主要新增模块。凡是候选节点声称 final，agents 必须辩论：

- 题目要求的 `target_slot` 是什么；
- 候选实际给出的 `candidate_slot` 是什么；
- `slot_match` 是否为 exact；
- 关系是否已经 complete；
- 如果不是 final，还缺哪一跳 `missing_hop`。

该模块主要修复旧版失败中的“过早终止”：

- spouse 被当成 child；
- town 被当成 borough；
- intermediate owner/location 被当成最终行政实体；
- 模型在第一跳就把 answer 字段填上，但节点文本只完成了桥接实体。

### 2.4 Atomic Quantity Closure

HotpotQA 中出现过一个副作用：题目表面说 country，但数据集实际答案是一个 county population 数值。严格槽位审计会把正确数值错误地重新打开。

因此 v8 加入一个协议级约束：当 QA 答案是明确出现在候选节点文本中的数量/年份/日期时，默认把它视为原子答案槽位，不因为实体词面歧义强行继续扩展。

这不是 bad-case 兜底，而是对 quantity/date QA 的通用终止性定义。

### 2.5 Contested Terminal Branch

硬删除被 terminality debate 否定的 final，会伤害短链数据集。v8 改为：

- 若 terminality debate 判定某 final 节点只是中间节点，则原候选变成 non-final，继续扩展；
- 同时保留一个低分 `contested_terminal` sibling；
- 最终路径辩论仍能看到这个被反驳过的原始立场；
- 如果后续扩展找不到更好答案，contested branch 仍有机会被最终 debate 选回。

这一点很重要：它让算法更像多 agent debate，而不是单向审计器。反方赢了局部辩论，但败方立场没有被从树中擦除，只是降权并留给终局辩论复审。

### 2.6 可消融开关

- `--ldt_no_terminality_debate`：关闭终止性辩论；
- `--ldt_no_contested_terminals`：关闭争议终点保留，退回硬删除式 terminality；
- `--ldt_use_contract`：根部 reasoning contract，可选；
- `--ldt_use_synthesis`：compatible synthesis，可选；
- `--ldt_use_slot_grounding`：终端 answer-slot grounding，可选。

当前默认主候选只开启 terminality debate、atomic quantity closure 和 contested terminal branch。

## 3. 真实轨迹观察

### 3.1 成功修复：spouse → child

MuSiQue 第 5 条旧版输出 `Daniel Webster`，停在 “Caroline LeRoy's spouse”。v8 terminality debate 判定：

- `target_slot`: child of Caroline LeRoy's spouse
- `candidate_slot`: spouse
- `slot_match`: too_narrow
- `missing_hop`: child of

树继续扩展后得到 `Fletcher Webster`，从错误修正为正确。

### 3.2 成功修复：town → borough

MuSiQue 第 12 条旧版输出 `Whiston`。v8 判定：

- 题目问 borough；
- 候选给的是 town；
- `slot_match=wrong_type`；
- 缺失 hop 为 borough。

后续扩展到 `Metropolitan Borough of Knowsley`，最终输出 `Knowsley`。

### 3.3 失败仍存在：证据选择和时态/版本事实

例如 “Stadio Ciro Vigorito 的球队所在 league” 一题，v8 可以正确把第一跳 `Benevento Calcio` 重新打开，但后续模型选择了 `Serie A`，而 gold 是 `Lega Pro Prima Divisione`。这类错误不是终止性问题，而是候选扩展阶段的证据版本选择问题。

### 3.4 失败仍存在：描述性占位答案

例如 “Adelphia Communications Corporation's successor in bankruptcy” 这种答案是关系描述而非具体公司名。当前 terminality debate 仍可能把它误判为 exact。下一轮应让 terminality debate 对“命名实体 vs 描述性占位”更敏感。

## 5. 当前选择

当前最适合作为主叙事版本的是：

```text
LDTD-v8: Contested Terminal Debate Tree
```

选择理由：

1. 明确体现多 agent debate，而不是多树投影或 ToT 搜索；
2. 对 q3-8B 和 q25-14B 都有 full50 宏平均提升；
3. 关键收益来自可解释的算法组件：terminality debate 修复过早终止；
4. contested terminal branch 解决了硬审计过度删除的问题；
5. 各组件边界清晰，便于后续消融。
